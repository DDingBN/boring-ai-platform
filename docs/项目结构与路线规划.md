# Boring AI Platform 项目结构与路线规划

## 1. 项目定位

Boring AI Platform 计划做成一个基于 monorepo 的 AI 学习平台。

这个项目的目标不只是简单调用一次 LLM API，而是搭建一个完整的 AI 应用，用来学习和实践以下能力：

- 与 LLM 对话
- 会话持久化
- 知识库与 RAG
- Embedding 与检索
- Tool calling
- Agent 执行
- Workflow 编排
- 前端控制台
- 运行日志与调试可视化

最终，这个项目应该成为一个本地优先的 AI workflow 和 agent 平台，用来系统学习真实 AI 应用的架构。

## 2. 当前项目结构

```txt
boring-ai-platform/
  apps/
    web/              # React + Vite 前端应用
    server/           # Express API 服务

  packages/
    shared/           # 前后端共享类型和轻量工具
    ai/               # LLM、Embedding、RAG、Tools、Agent、Workflow 逻辑
    database/         # 数据库客户端、schema、repositories

  package.json
  pnpm-workspace.yaml
  turbo.json
  tsconfig.base.json
```

当前结构适合继续发展。后续要重点保持模块边界清晰：

- `apps/web` 负责用户交互和可视化。
- `apps/server` 负责 HTTP API、请求校验、服务编排，以及后续认证。
- `packages/shared` 负责前后端共享的数据契约。
- `packages/ai` 负责 AI 相关运行时逻辑。
- `packages/database` 负责数据持久化和数据库访问。

不要把 AI 逻辑或数据库逻辑直接堆进 `apps/server`。server 应该主要做 API 层编排。

## 4. 各模块职责

### apps/web

前端应该成为整个项目的主要操作界面。

必要页面：

- `/chat`：与模型对话，选择模型，配置参数。
- `/knowledge`：管理知识库，上传文档，查看文档切片。
- `/workflows`：创建、编辑、保存、运行 workflow。
- `/runs`：查看 workflow 运行记录、prompt、检索片段、工具调用和错误。

建议具备的 UI 能力：

- 聊天消息列表
- assistant 流式响应
- 会话侧边栏
- 模型选择器
- 知识库选择器
- 文档上传
- Workflow 图编辑器
- 运行日志面板

### apps/server

后端应该提供稳定 API，并负责协调各个 package。

推荐 API：

- `GET /health`
- `POST /api/chat`
- `GET /api/conversations`
- `POST /api/conversations`
- `GET /api/conversations/:id/messages`
- `POST /api/knowledge-bases`
- `POST /api/documents`
- `POST /api/workflows`
- `POST /api/workflows/:id/run`
- `GET /api/runs/:id`

server 不应该包含底层 LLM、RAG、Embedding 或 Prisma 实现细节。

### packages/shared

shared 应该只放共享契约，不放业务实现。

推荐类型：

- `ChatMessage`
- `Conversation`
- `ModelConfig`
- `KnowledgeBase`
- `Document`
- `DocumentChunk`
- `WorkflowGraph`
- `WorkflowNode`
- `WorkflowRun`
- `ToolCall`
- API request 和 response 类型

### packages/ai

这是项目最核心的学习模块。

必要能力：

- LLM provider 抽象
- 文本生成
- 流式文本生成
- Embedding 生成
- 文本切片
- 向量检索
- RAG prompt 组装
- Tool registry
- Tool calling
- Agent 执行循环
- Workflow 节点执行

Provider 相关代码要放在接口后面，避免业务层直接依赖某一个模型供应商。这样后续可以支持 OpenAI-compatible API、本地模型或其他 provider。

### packages/database

database package 应该统一管理持久化逻辑。

推荐基础能力：

- Prisma schema
- Prisma client singleton
- Repository 函数
- 数据库 migrations
- demo seed 脚本

建议数据模型：

- `Conversation`
- `Message`
- `KnowledgeBase`
- `Document`
- `DocumentChunk`
- `Embedding`
- `Workflow`
- `WorkflowRun`
- `WorkflowRunStep`
- `ToolCall`
- `ModelConfig`

认证可以后续再加。学习阶段先做单用户本地模式就够用。

## 6. 开发路线

### Phase 1：基础设施

目标：让仓库成为一个可靠的开发基础。

任务：

- 保持 `pnpm typecheck` 通过。
- 保持 `pnpm build` 通过。
- 添加 `.env.example`。
- 添加基础 README 启动说明。
- 把 placeholder lint 脚本替换成真实 lint 命令。

状态：部分完成。

### Phase 2：最小 Chat 闭环

目标：把 frontend、server、shared types 和 AI package 串起来。

链路：

```txt
web chat input
  -> POST /api/chat
  -> server chat service
  -> packages/ai LLM client
  -> assistant response
  -> web message list
```

任务：

- 在 `packages/shared/src/chat.ts` 中添加 chat request/response 类型。
- 添加 LLM provider interface。
- 添加 OpenAI-compatible provider。
- 添加 `POST /api/chat`。
- 添加简单 chat UI。
- 处理 API key 缺失时的错误提示。

这是下一个最重要的里程碑。

### Phase 3：数据持久化

目标：保存会话和消息。

任务：

- 添加 Prisma。
- 添加 `DATABASE_URL`。
- 添加 `Conversation` 和 `Message` 模型。
- 添加 repositories。
- 保存 user 和 assistant 消息。
- 在 web 中加载历史会话。

### Phase 4：知识库与 RAG

目标：支持基于文档的问答。

链路：

```txt
upload document
  -> extract text
  -> chunk text
  -> generate embeddings
  -> store chunks
  -> user asks question
  -> retrieve relevant chunks
  -> build prompt with context
  -> answer with citations
```

任务：

- 添加知识库模型。
- 添加文档上传 API。
- 添加 chunker。
- 添加 embedder。
- 添加 retriever。
- 添加 RAG pipeline。
- 添加前端知识库页面。
- 在回答中展示检索到的 chunks。

### Phase 5：Workflow Runtime

目标：把 `WorkflowGraph` 从单纯类型变成可执行系统。

支持的节点类型：

- `start`
- `prompt`
- `llm`
- `rag`
- `tool`
- `end`

任务：

- 定义节点配置类型。
- 添加 workflow executor。
- 运行前校验 graph。
- 按顺序执行节点。
- 保存节点输入、输出、错误和耗时。
- 添加 workflow run API。

### Phase 6：Workflow UI

目标：用可视化方式创建和运行 workflows。

任务：

- 添加 workflow 列表页。
- 添加 workflow 编辑页。
- 使用 React Flow 这类图编辑器库。
- 添加节点配置面板。
- 保存和加载 workflow graph。
- 从 UI 运行 workflow。
- 展示执行状态和日志。

### Phase 7：Tools 与 Agents

目标：支持多步骤 AI 行为。

任务：

- 添加 tool registry。
- 添加内置 tools。
- 添加 tool call 日志。
- 添加 agent loop。
- 添加最大执行步数限制。
- 添加安全和超时处理。
- 在前端展示 tool calls。

### Phase 8：质量与可观测性

目标：让项目更容易调试和维护。

任务：

- 添加 `packages/ai` 单元测试。
- 添加 `apps/server` API 测试。
- 添加 workflow executor 测试。
- 添加 RAG pipeline 测试。
- 添加结构化日志。
- 记录 model、tokens、latency、retrieved chunks 和 tool calls。
- 添加 seed/demo 数据。

## 8. 完成标准

当项目具备以下能力时，可以认为它已经是一个完整的 AI 学习平台：

- 能通过一个清晰的 setup 流程启动。
- 能从 web app 与 LLM 对话。
- 能持久化会话和消息。
- 能从上传文档创建知识库。
- 能基于检索到的文档 chunks 回答问题。
- 能创建并运行 workflow graph。
- 能展示 prompt、模型输出、tool calls、检索结果和错误。
- 能保持 `pnpm typecheck` 和 `pnpm build` 通过。
- 对 RAG 和 workflow 执行有重点测试覆盖。

## 9. 立即下一步

建议下一步进入 Phase 2：最小 Chat 闭环。

先添加：

- `packages/shared/src/chat.ts` 中的 chat 类型
- `packages/ai/src/llm` 中的 LLM client interface
- OpenAI-compatible provider
- `apps/server` 中的 `POST /api/chat`
- `apps/web` 中的简单 chat 页面

这一步会第一次真正打通项目的几层结构，让项目从“框架”变成可运行的 AI 应用闭环。
