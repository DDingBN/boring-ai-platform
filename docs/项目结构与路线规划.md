# Boring AI Platform 项目结构与路线规划

> 本文同时描述项目定位与演进方向。仓库中已经实现的能力以
> [当前实现状态](./current-status.md) 为准；本文出现的 Run、RAG、Workflow、Tool 和 Agent
> 等内容大多仍是目标设计。

## 1. 项目定位

Boring AI Platform 是一个本地优先的 AI workflow 和 agent 学习平台。

它的目标不是做一个普通聊天 demo，也不是先做完整管理后台，而是通过一个可运行、可观察、可扩展的 AI 应用，系统练习真实 AI 产品中的核心能力：

- LLM 对话与流式响应
- 会话和运行记录持久化
- 知识库、Embedding、检索和 RAG
- Tool calling 和 agent 执行循环
- Workflow 编排和运行调试
- 前端控制台和执行过程可视化

当前阶段最重要的方向是先打通最小闭环，并为后续 RAG、workflow、tools 和 agents 预留统一的运行时底座。

## 2. 设计主线

后续设计优先围绕以下原则推进：

1. **运行记录优先**
   从最小 chat 阶段开始设计 `Run`、`RunStep`、`TraceEvent`。chat、RAG、tool call、workflow 和 agent 都应该写入同一套运行记录，而不是每个功能各自记录日志。

2. **契约先行**
   `packages/shared` 只放前后端共享契约。请求 DTO、响应 DTO、持久化实体、流式事件要分开定义。仅有 TypeScript 类型不够，API 边界建议配合 runtime schema 校验。

3. **流式协议可复用**
   Chat 的流式响应不要做成一次性实现。建议使用 SSE，并定义可复用事件，例如 `message.delta`、`run.step.started`、`retrieval.completed`、`tool.call.completed`、`error`。

4. **模块边界清晰**
   `apps/server` 负责 API、校验、服务编排和认证入口；LLM、RAG、tools、agents、workflow executor 放在 `packages/ai`；数据库 schema、client 和 repositories 放在 `packages/database`。

5. **前端围绕真实工作流**
   首页和页面不要过早堆 mock 指标。早期前端应优先做成可用的 AI workbench：对话、运行记录、调试面板、知识库和 workflow 逐步展开。

## 4. 当前项目结构

```txt
boring-ai-platform/
  .env.example        # 无密钥的本地配置模板
  package.json        # 根命令与共享开发依赖
  pnpm-workspace.yaml # workspace 范围
  turbo.json          # dev/build/lint/test/typecheck 任务

  apps/
    web/              # React 19 + Vite 8 + Ant Design 6 前端
      src/
        components/   # 布局与菜单组件
        data/         # 菜单、路由注册表和错误页元数据
        pages/        # home、chat、login、error 页面
        router/       # React Router 路由定义
    server/           # Express 5 API 服务
      src/
        app.ts        # 创建应用并注册基础 middleware/routes
        config/env.ts # 根 .env 加载与 Server 配置校验
        chat/index.ts # 尚未挂载的 Chat Router 占位
        middleware/   # request ID 与统一错误处理
        index.ts      # 加载配置并监听端口

  packages/
    shared/           # 当前导出 Chat、Workflow、Knowledge 等 TypeScript 类型
    ai/               # 当前只有模型请求相关类型占位
    database/         # 当前只有数据库配置和健康状态类型占位

  docs/               # 架构、模块和路线规划
```

这个结构可以继续使用。当前没有 `apps/server/src/routes`、`services`、Prisma schema、数据库
client 或 AI provider；相关结构在模块文档中属于后续建议，不是现有文件。

## 6. 近期里程碑

下一步建议进入 **最小 Chat + Run Inspector 闭环**。

目标链路：

```txt
web chat input
  -> POST /api/chat or SSE /api/chat/stream
  -> server chat service
  -> packages/ai LLM provider
  -> Run / RunStep / TraceEvent
  -> assistant response
  -> web message list + run inspector
```

最小交付物：

- `packages/shared` 定义 chat request、chat response、stream event、API error。
- `packages/ai` 添加 LLM provider interface 和 OpenAI-compatible provider。
- `apps/server` 添加 `POST /api/chat`，并预留 SSE streaming 结构。
- `apps/web` 将 mock chat 替换为真实 API 调用。
- 前端 chat 页面增加调试面板，展示请求、模型配置、耗时、错误和原始响应。
- API key 缺失时返回明确错误，不在前端泄露密钥。

其中 Shared 目前已有一版 `ChatMessage`、`ChatRequest` 和 `ChatResponse` 类型，但它们仍是占位契约；
Server 的固定响应 Chat Router 也尚未挂载，Web Chat 仍只在浏览器中生成 Mock 回复。

## 7. 完成标准

当项目具备以下能力时，可以认为它已经从脚手架进入可用的 AI 学习平台阶段：

- 能通过清晰的 setup 流程启动 web 和 server。
- 能从 web app 与 LLM 对话，并支持后续流式输出。
- 能记录每次 chat、RAG、tool、workflow 的运行过程。
- 能持久化会话、消息、运行记录和关键调试信息。
- 能从上传文档创建知识库，并基于检索 chunks 回答问题。
- 能创建、保存、运行 workflow graph。
- 能展示 prompt、模型输出、检索结果、tool calls、耗时和错误。
- `pnpm typecheck`、`pnpm build`、核心单元测试保持通过。

## 8. 暂缓内容

以下内容不适合作为早期重点：

- 多用户团队管理、邀请成员和复杂权限。
- 大而全的首页 dashboard。
- 没有真实数据来源的调用量、成功率、任务统计。
- 过早设计复杂 agent UI。
- 在 workflow executor 稳定前投入大量图编辑器细节。
